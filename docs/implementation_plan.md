# Kansu: infinite scroll archiver 実装計画

## 1. はじめに

本ドキュメントは、ブラウザ拡張機能「Kansu」の開発における実装計画を定義します。
目的：無限スクロールページからのデータ抽出、管理、エクスポート機能の実現。

## 2. セットアップフェーズ

### 2.1. 開発環境の準備

- Node.jsとpnpmのインストール確認
- Gitのセットアップ

### 2.2. プロジェクトの初期化

- `pnpm create wxt` コマンドを実行し、WXTプロジェクトをセットアップする。
  - テンプレート: `react`
  - パッケージマネージャー: `pnpm`

### 2.3. ライブラリの導入

プロジェクト初期化後、以下のライブラリを導入する。

- **コード品質 (Biome)**
  - プロジェクトにBiomeを導入する。
  - 適宜フォーマッターやリンターのルールを設定する。
- **ドキュメント規約 (MarkdownLint)**
  - プロジェクトにMarkdownLintを導入する。
  - 適宜ドキュメント規約のルールを設定する。
- **UIコンポーネント (Shadcn/ui)**
  - wxtとの構成を考慮したShadcn/uiの初期設定を行う。ref: <https://github.com/wxt-dev/examples/tree/main/examples/react-shadcn>
  - 必要なコンポーネント（`button`, `input`, `toast`など）は必要になったタイミングで個別に追加する。
- **テストフレームワーク (Vitest)**
  - プロジェクトにVitestを導入する。
- **E2Eテストフレームワーク (Playwright)**
  - WXTの公式ガイドに従い、Playwrightを導入してE2Eテスト環境を構築する。
- **データストレージ (Dexie.js)**
  - ライブラリをインストールする。
- **状態管理 (Zustand)**
  - ライブラリをインストールする。

その他、必要になったタイミングでそれに応じたライブラリを導入する。

## 3. 実装フェーズ

WXTが提供するVitest連携 (`wxt/testing`) を活用し、テストドリブン開発（TDD）で実装を進める。

### 3.1. テスト戦略

テスト環境のセットアップは、WXTの公式ドキュメントや関連情報を参考に進める。

- **ユニットテスト**: Vitestを使用し、個別の関数やReactコンポーネント、カスタムフックのロジックをテストする。
  - `wxt/testing`が提供する`fake browser` APIを利用し、ブラウザ拡張機能APIに依存する処理（`browser.storage`など）をモックなしでテストする。
- **E2Eテスト**: （将来的な検討事項）Playwrightなどを導入し、実際のブラウザ環境でユーザー操作をシミュレートするテストを検討する。

### 3.2. 実装ステップ

1. **データ構造の定義**: `types.ts`などに、アプリケーション全体で利用するデータ構造（設定、抽出データなど）の型を定義する。
2. **Background Scriptの実装 (TDD)**
    - `vitest`でテストファイルを作成。
    - Dexie.jsを用いたIndexedDBへのデータ保存、取得、検索、削除処理をテストファーストで実装する。
    - `fake browser` APIを活用し、`browser.runtime.onMessage`などのイベントリスナーの動作をテストする。
3. **Content Scriptの実装 (TDD)**
    - データ抽出ロジックをテストする。DOM操作が絡む部分は、`happy-dom`や`jsdom`をテスト環境に導入し、HTML文字列からDOMを構築してテストを行う。
    - バックグラウンドスクリプトとのメッセージ送受信ロジックをテストする。
4. **UIコンポーネントの実装**
    - VitestとReact Testing Libraryを使用し、コンポーネントのレンダリング、インタラクション、状態変更をテストする。
5. **E2Eテストによる動作確認**
    - Playwrightを使用し、実際のブラウザでポップアップの表示、オプションページの操作、コンテンツスクリプトの動作など、一連のユーザーフローをテストする。
6. **各エントリーポイントの結合**
    - 実装した各モジュール（Background, Content, UI）をエントリーポイントで結合し、全体の動作を確認する。

## 4. デプロイ

- `pnpm build`によるビルド
- Chrome Web Storeへの公開
