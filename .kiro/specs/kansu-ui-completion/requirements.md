# Kansu UI完成機能 要件定義書

## はじめに

本要件定義書は、ブラウザ拡張機能「Kansu (巻子)」の残りUI機能の実装要件を定義します。Phase 1-3（データ構造、Background Service、Content Script）が完了した現在、Phase 4として以下のUI機能の実装が必要です：

- メインUI（データ表示・検索・管理機能）
- Popup UI（表示切り替え・基本操作）
- Options UI（設定管理・データ管理）
- 共通UIコンポーネント（Toast、Modal等）

## 要件

### 要件1: メインUI機能

**ユーザーストーリー:** 拡張機能ユーザーとして、ウェブページ上に注入されるメインUIで、抽出されたデータの検索・表示・管理を効率的に行いたい。

#### 受入基準

1. WHEN ユーザーが対象サイトを閲覧 THEN メインUIがページ内に注入される
2. WHEN メインUIが表示される THEN 抽出されたデータが一覧表示される
3. WHEN ユーザーが検索キーワードを入力 THEN インクリメンタル検索が実行される
4. WHEN ユーザーが検索対象フィールドを選択 THEN 指定フィールドのみが検索対象となる
5. WHEN ユーザーがソート項目を選択 THEN データが昇順/降順でソートされる
6. WHEN データが大量にある場合 THEN ページネーション機能が動作する
7. WHEN ユーザーがページあたり表示件数を変更 THEN 表示が更新される
8. WHEN 日本語の表記揺れがある場合 THEN 正規化して検索される（全角/半角、ひらがな/カタカナ）
9. WHEN ユーザーがメインUIを閉じる THEN UIが非表示になる
10. WHEN メインUIが表示される THEN ページのスタイルに影響されない独立したスタイルが適用される

### 要件2: Popup UI機能

**ユーザーストーリー:** 拡張機能ユーザーとして、ツールバーアイコンをクリックした際に表示されるポップアップで、メインUIの表示切り替えと基本的な操作を素早く行いたい。

#### 受入基準

1. WHEN ユーザーが拡張機能のツールバーアイコンをクリック THEN ポップアップUIが表示される
2. WHEN ポップアップが表示される THEN 現在のタブでのメインUI表示状態（表示/非表示）が確認できる
3. WHEN ユーザーがメインUI表示切り替えボタンをクリック THEN 現在のタブでメインUIの表示/非表示が切り替わる
4. WHEN ユーザーがオプション設定リンクをクリック THEN Options UIページが新しいタブで開く
5. WHEN ポップアップが表示される THEN 現在のタブで抽出されたデータの件数が表示される
6. WHEN 現在のタブがKansuの対象サイトでない場合 THEN 「このサイトは対象外です」というメッセージが表示される
7. WHEN ポップアップが表示される THEN 統一されたデザインシステム（Shadcn/ui）が適用されている

### 要件3: Options UI機能

**ユーザーストーリー:** 拡張機能ユーザーとして、詳細な設定画面で抽出ルールの管理とデータの管理を行いたい。

#### 受入基準

1. WHEN ユーザーがOptions UIにアクセス THEN サービス設定の一覧が表示される
2. WHEN ユーザーが新しいサービス設定を追加 THEN サービス名、対象URL、抽出ルールを設定できる
3. WHEN ユーザーが既存のサービス設定を編集 THEN 設定内容を変更して保存できる
4. WHEN ユーザーがサービス設定を削除 THEN 確認ダイアログ後に設定が削除される
5. WHEN ユーザーがフィールド設定を追加/編集 THEN フィールド名、CSSセレクタ、タイプ（text/linkUrl/imageUrl/regex）を設定できる
6. WHEN ユーザーがregexタイプのフィールドを設定 THEN 正規表現パターンを入力できる
7. WHEN ユーザーがデータエクスポート機能を使用 THEN 指定したサービスのデータをJSONファイルでダウンロードできる
8. WHEN ユーザーがデータインポート機能を使用 THEN JSONファイルを選択してデータを復元できる
9. WHEN ユーザーがデータ削除機能を使用 THEN 指定したサービスのデータを確認ダイアログ後に削除できる
10. WHEN Options UIが表示される THEN 統一されたデザインシステム（Shadcn/ui）が適用されている

### 要件4: 共通UIコンポーネント機能

**ユーザーストーリー:** 拡張機能ユーザーとして、各UI間で一貫した操作体験とフィードバックを得たい。

#### 受入基準

1. WHEN UI操作でエラーが発生 THEN 適切なエラーメッセージがToast通知で表示される
2. WHEN データ操作が完了 THEN 成功メッセージがToast通知で表示される
3. WHEN 削除操作を実行 THEN 確認Modalダイアログが表示される
4. WHEN Modalダイアログが表示される THEN 背景がオーバーレイされる
5. WHEN Toast通知が表示される THEN 一定時間後に自動で消える
6. WHEN 各UIコンポーネントが表示される THEN 統一されたデザインシステム（Shadcn/ui）が適用されている

### 要件5: UI統合・通信機能

**ユーザーストーリー:** 拡張機能ユーザーとして、各UI間の連携がスムーズに動作し、データの整合性が保たれることを期待する。

#### 受入基準

1. WHEN Popup UIでメインUI表示を切り替え THEN Content ScriptのメインUIが即座に反応する
2. WHEN Options UIで設定を変更 THEN Content Scriptの抽出動作が新しい設定で動作する
3. WHEN メインUIでデータを表示 THEN Background Serviceから最新のデータが取得される
4. WHEN Options UIで設定を変更 THEN Background Serviceを通じてIndexedDBに永続化される
5. WHEN Popup UIでデータ件数を表示 THEN 最新のIndexedDBデータが反映される
6. WHEN データインポート/エクスポート操作を実行 THEN データの整合性が検証される
7. WHEN 複数のタブで同時に操作 THEN データの競合状態が適切に処理される
8. WHEN UI操作でエラーが発生 THEN データの不整合状態にならない

### 要件6: レスポンシブ・アクセシビリティ機能

**ユーザーストーリー:** 拡張機能ユーザーとして、様々な画面サイズや操作環境で快適に利用したい。

#### 受入基準

1. WHEN メインUIが小さな画面で表示される THEN レスポンシブデザインで適切に表示される
2. WHEN ユーザーがキーボードで操作 THEN 適切なフォーカス管理とキーボードナビゲーションが動作する
3. WHEN スクリーンリーダーを使用 THEN 適切なARIAラベルとセマンティックHTMLが提供される
4. WHEN 高コントラストモードを使用 THEN 視認性が保たれる
5. WHEN メインUIが他のページ要素と重複 THEN 適切なz-indexで前面に表示される
